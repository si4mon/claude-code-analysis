Claude Code Agent Runtime 的 Harness Engineering 分析

核心论点：Claude Code 的稳定性不是来自“更强模型”本身，而是来自一个层层保护的 runtime harness： 主循环编排、上下文压缩恢复、动态工具暴露、结构化 I/O 与持久化链路共同作用，把复杂任务从“容易断裂”变成“可恢复推进”。

1. 叙事起点：为什么这套 runtime 值得分析

在复杂 agent 场景中，失败通常不发生在“模型不会答”，而发生在“工具调用中断、上下文爆炸、状态丢失、恢复失败”。 这份代码的价值在于：它把这些失败点显式工程化。主轴是 query.ts，它不是一次请求函数，而是一个可恢复的 agent 回合引擎。

2. 证据链：与 Harness 6 组件逐一对齐

2.1 推理引擎：动态选模 + 预算 + 回退

  • 运行时根据权限模式与上下文规模选模（getRuntimeMainLoopModel）。
  • 发生 fallback 时，清理 orphan 的中间状态，再重试，避免 tool_use/tool_result 链被污染。
  • task_budget 与 token_budget 双预算并行，覆盖“整轮任务”而非单请求。

技术价值：把模型波动从“系统中断”降级为“可恢复分支”。

2.2 计划与编排：主循环状态化 + 多停止条件

  • while(true) + State 聚合跨轮状态，transition.reason 记录继续原因。
  • 停止条件覆盖完成、hook 阻断、max turns、abort、预算到达、恢复失败等多个出口。
  • 对 stop_reason === 'tool_use' 不可靠性做运行时补偿（以实际 tool_use block 为准）。

技术价值：把“何时停、为何继续”从隐式行为变成可观测语义。

2.3 工具注册表：可见性前置治理 + 动态暴露

  • tools.ts 统一组装 built-in + MCP 工具池，并做 deny 规则过滤。
  • services/api/claude.ts 支持 ToolSearch 与 deferred loading：只把“已发现”的 deferred tools 真正发给 API。
  • Tool.ts 定义统一契约（schema、并发安全、权限检查、结果大小限制等）。

技术价值：降低提示噪音与误调用，支持工具规模增长。

2.4 内存与上下文：多层压缩管线 + 恢复回路

  • 同一轮中组合使用 toolResultBudget → snip → microcompact → collapse → autocompact。
  • 遇到 413 / media / max_output 等错误时，先 withheld，再尝试恢复（collapse drain / reactive compact / escalation）。
  • memory 与 skill prefetch 并行于流式生成，后置注入，减少阻塞。

技术价值：在“信息保真”和“token 可控”之间做动态平衡，而不是粗暴截断。

2.5 状态与持久化：JSONL 事件流 + 恢复一致性

  • sessionStorage.ts 维护 transcript / subagent 路径与 parentUuid 链一致性。
  • 明确区分 transcript message 与 progress message，避免恢复链污染。
  • history.ts 采用 lock + append，且大文本通过 hash 外置到 paste store。

技术价值：把“可恢复性”变成可工程化的底座能力。

2.6 结构化输入输出：强约束 + 可恢复错误协议

  • 工具输入统一 inputSchema.safeParse；失败返回结构化 tool_result(is_error)。
  • API 前统一消息归一化、tool-pair 修复、字段剥离，降低线上 400/格式错误。
  • 权限、hooks、校验在执行前完成，确保 I/O 不只是“格式正确”，还“语义可执行”。

技术价值：输出能被后续步骤直接消费，减少链式故障传播。

3. 反面证据：当前仍相对缺少的 Harness 实践

3.1 缺少显式状态机定义层

当前实现接近状态机，但仍以多分支 continue 维护复杂流转。随着恢复路径增长，分支交互风险上升。

影响：变更验证成本升高，易出现边界条件回归。

3.2 持久化路径存在较多 best-effort 处理

历史/会话部分写入失败采用日志记录后继续（含 fire-and-forget），利于可用性但削弱强一致保障。

影响：少量场景下可能出现“用户感知已保存，实际未完整落盘”的恢复缺口。

3.3 多 agent 的全局调度语义仍偏轻量

目前已有作用域隔离与队列快照，但缺少更上层统一调度（公平性/背压/抢占）框架。

影响：高并发长任务场景下，尾延迟与资源争抢风险增加。

4. 综合评估：技术价值与成熟度

组件 技术价值 成熟度 可吸收点
推理引擎动态选模 + 回退 + 预算高先恢复再报错
计划编排多停止条件 + hooks高把继续/终止理由显式化
工具注册defer + ToolSearch + deny 前置高分离“工具存在”与“工具暴露”
上下文管理分层压缩 + 恢复回路高采用策略管线而非单摘要
状态持久化链路一致性强中高优先保证可恢复主链
结构化 I/O输入校验到输出协议闭环高把错误也做成可消费结构

5. 技术启发（结论）

这份仓库给出的最大启发是：稳定 agent 的关键不是单个“智能”模块，而是 runtime harness 的联合作用。 如果你的目标是“可靠地完成复杂任务”，建议按以下顺序建设：

  1. 先实现可恢复主循环（含停止条件与恢复分支）；
  2. 再实现结构化工具契约（schema/权限/错误协议）；
  3. 随后引入分层上下文压缩与预算治理；
  4. 最后完善多 agent 调度与长期记忆质量闭环。

这条路径的本质是：先让系统“不会轻易坏”，再让系统“越来越聪明”。
